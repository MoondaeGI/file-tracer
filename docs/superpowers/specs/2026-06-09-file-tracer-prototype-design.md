## 1. 목적과 포지셔닝

사내 "위험 파일"을 파일 지문으로 추적하는 시스템의 **첫 프로토타입**. 이 단계의
목표는 제품 완성이 아니라 **두 가지 가설 검증**이다.

1. 파일 지문(SHA-256 정확 매칭 + ssdeep fuzzy 유사도)이 기대대로 동작하는가 — 같은 파일은
   정확히 일치하고, 내용을 조금 수정하면 합리적인 유사도(%)가 나오는가.
2. **파일 본문을 서버에 올리지 않고 지문(수백 바이트)만으로** 동일한 트레이스가
   성립하는가.

포지셔닝은 "법적 증거 시스템"이 아니라 **억지·감사 도구**다. 증거능력은 부수효과로만
취하며(서버 타임스탬프 + 해시체인), 실제 사내 배포 시에는 별도의 법무·노사 검토가
전제된다(이번 범위 밖).

## 2. 범위

### 이번 단계에 포함

- `server/` 폴더, 전부 Python.
- 파일 지문 생성·등록·매칭 API.
- 등록 결과·매칭 결과를 보여주는 단일 웹페이지.
- 서버 타임스탬프 + append-only 해시체인(변조탐지 장부).

### 이번 단계에서 제외 (다음 단계로 명시 이월)

- OS 파일 이벤트 감시 클라이언트(`client/`, watchdog) — 서버 계약이 굳은 뒤 부착.
- 본문 텍스트 추출 + 문서 단위 유사도 지문(docx/pdf 파싱 후 지문) — 무겁고 자주
  갱신되므로 서버에 둘 예정이나 프로토타입엔 미포함.
- 본문 업로드 게이트(워치리스트 ∩ 서버가 처음 보는 SHA일 때만 1회 업로드).
- 엣지케이스 정책: 암호화(엔트로피 플래그 + 정확 SHA), 초대용량(스트리밍 해시 +
  사이즈 캡 + 샘플 해시).
- 프로세스 귀속(process_name) / 다운로드·업로드 방향(direction) — ETW·커널 영역.
  과거 스펙에서 ETW는 고부하 시 이벤트 드롭이 잦았던 경험이 있어 채택하지 않는다.

## 3. 아키텍처

```
server/                     (전부 Python)
  FastAPI + uvicorn         HTTP/JSON API
  fingerprint module        hashlib(SHA-256) + ppdeep(ssdeep 계열 fuzzy)
  repository (SQLite)        얇은 추상화 — in-memory로 교체 가능
  static web page           업로드 폼 + 결과 표 + 등록 목록
```

> **fuzzy 알고리즘 변경 이력(2026-06-09):** 원안은 TLSH(`python-tlsh`)였으나, 그
> 패키지가 Windows에서 소스 빌드가 깨져(어느 Python 버전이든 `WinFunctions.h` 누락)
> 설치 불가였다. 순수 파이썬 `ppdeep`(ssdeep 계열, 컴파일 불필요)로 교체한다. ssdeep은
> `compare(a, b)`가 **0~100 유사도를 직접** 반환하므로 TLSH diff→similarity 환산식이
> 불필요해지고, 그 환산식의 포화 문제(앞선 검증 라운드의 🔴 지적)도 함께 사라진다.
> 지문 필드명은 `tlsh` → `fuzzy_hash`로 바꾼다. (TLSH를 고른 본래 이유는 클라이언트
> 이식성이었으나 서버 프로토타입엔 무관하며, 클라 알고리즘은 향후 재론.)

- **언어/스택**: Python. 서버 FastAPI. (실제 배포용 감시 에이전트는 향후 Go/C# 등
  네이티브로 갈아끼울 수 있으며, 그때도 서버·API 계약·웹페이지는 그대로 재사용한다.
  교체되는 것은 클라이언트의 OS 감시 층뿐이다.)
- **저장소**: SQLite(표준 라이브러리 `sqlite3`). 재시작 후에도 등록 지문·해시체인이
  유지된다. repository 인터페이스로 추상화하여 in-memory 구현으로 교체 가능.

## 4. API 계약

### 4.1 `POST /api/fingerprints` — 지문 등록 + 매칭

입력은 **두 모드**를 지원하며, 두 모드 모두 동일한 매칭 로직을 타고 동일한 응답을
돌려준다.

- **(a) 파일 본문 업로드** (`multipart/form-data`): 서버가 바이트를 읽어 지문을
  계산한다. *지문 알고리즘 검증·기준 파일 등록용.* → 레코드는 `record_kind=baseline`.
- **(b) 지문만** (`application/json`): 본문 없이 이미 계산된 지문과 메타데이터만
  전송한다. *실제 트레이스 경로 — 향후 클라이언트가 사용할 계약.* → 레코드는
  `record_kind=event`.

**중요 — 모드 (b) 지문의 신뢰 한계(R8 부활):** 모드 (b)에서 서버는 클라가 보낸
`sha256`/`fuzzy_hash`가 실제 그 파일에서 나온 값인지 **본문이 없으므로 검증할 수 없다.**
따라서 "(a)가 계산한 지문을 (b)로 되먹여 같은 결과가 나온다"는 것은 동어반복이며 아무
가설도 증명하지 못한다. 가설 2("본문 없이 트레이스 성립")가 실제로 묻는 것은 *서로
다른 주체·구현이 같은 파일에서 같은 지문을 재현하는가*이다 — 이건 §9 시나리오 4에서
독립 재계산으로 검증한다. 이 한계는 §10에 명시한다.

(b) 요청 본문 예:

```json
{
  "sha256": "a3f9c2...",
  "fuzzy_hash": "48:abcd...:efgh...",
  "size": 48210,
  "name": "설계도.dwg",
  "host": "PC-01",
  "user": "kim",
  "event_type": "modify"
}
```

응답 예 (두 모드 공통):

```json
{
  "id": 42,
  "sha256": "a3f9c2...",
  "fuzzy_hash": "48:abcd...:efgh...",
  "registered_as": "file_042",
  "record_kind": "event",
  "matches": [
    { "id": 17, "name": "설계도.dwg",    "match_type": "exact", "similarity": 100 },
    { "id": 31, "name": "설계도_v2.dwg", "match_type": "fuzzy", "similarity": 87 }
  ],
  "server_timestamp": "2026-06-09T13:55:00+09:00"
}
```

`similarity`는 ssdeep `compare()`가 직접 주는 0~100 값이다(exact는 항상 100). 별도
환산식·`tlsh_diff` 필드는 없다.

### 4.2 `GET /api/fingerprints` — 등록 목록 조회

등록된 지문·이벤트 레코드 목록을 반환한다(웹페이지 렌더링용). 페이징은 프로토타입
범위 밖 — 최근 N개 반환.

**인증 없음은 의도적 이월이다.** 이 GET은 host/user/name/event_type을 평문으로
노출하므로("누가 언제 어떤 위험 파일을 만졌나"는 그 자체로 민감 정보) 실제 배포 전
인증·접근통제가 필요하다. 프로토타입에선 단일 로컬 검증이 목적이라 넣지 않는다(§10).

## 5. 데이터 모델 (레코드)

forward-compatible: 향후 클라이언트·네이티브 에이전트가 채울 필드를 미리 nullable로
예약한다. 프로토타입은 비워두거나 일부만 채운다.

| 필드 | 설명 | 프로토타입 |
|---|---|---|
| `id` | 자동 증가 PK | 채움 |
| `sha256` | 정확 해시 | 채움 |
| `fuzzy_hash` | ssdeep(ppdeep) fuzzy 해시 | 채움(빈 입력 시 null) |
| `size` | 바이트 크기 | 채움 |
| `name` | 파일명 | 채움 |
| `host` | 보고 호스트 | 채움(미상 시 null) |
| `user` | 사용자 | 채움(미상 시 null) |
| `event_type` | upload/modify/move/delete 등 | 채움 |
| `record_kind` | `baseline`(기준 위험파일·모드 a) / `event`(들어온 보고·모드 b) | 채움 |
| `process_name` | 행위 프로세스 | null (확장 예약) |
| `direction` | download/upload/local | null (확장 예약) |
| `source_hint` | 경로 휴리스틱(gdrive_sync 등) | null (확장 예약) |
| `server_timestamp` | **서버가** 찍는 시각 | 채움 |
| `prev_hash` | 직전 레코드의 record_hash | 채움 |
| `record_hash` | 이 레코드 정규화 직렬화의 해시 | 채움 |

## 6. 매칭 로직

**매칭 대상 = `record_kind=baseline` 레코드들만.** 새 지문이 들어오면 기존 *baseline*
레코드들과만 비교한다(전체 레코드가 아님 — 이벤트끼리는 비교하지 않는다). 이로써
이벤트가 누적돼도 매칭 표가 폭발하지 않고, "추적 기준이 무엇인가"가 데이터로 명확해진다.
**자기 자신은 제외한다**(방금 등록한 레코드가 100%로 잡히는 노이즈 방지).

1. `sha256` 동일 → `match_type: exact`, `similarity: 100`.
2. 아니면 ssdeep `compare(incoming.fuzzy_hash, baseline.fuzzy_hash)`로 **0~100 유사도를
   직접** 얻는다(별도 환산식 없음 — 이 값이 곧 raw 신호).
   - **매칭 포함 임계치(코딩 전 확정)**: `similarity ≥ 50`인 것만 `matches`에 포함하고
     `match_type: fuzzy`로 표시한다.
3. **fuzzy_hash=null 처리**: 입력이 비어 있어(0바이트) ppdeep이 의미 있는 지문을 못
   만들면 `fuzzy_hash=null`로 저장한다. null 지문은 fuzzy 매칭 대상에서 제외하며(null
   끼리·null 대 값 비교 안 함), 그 레코드는 SHA exact 매칭만 가능하다.
4. 매칭 결과는 `similarity` 내림차순 정렬(exact가 항상 위).

## 7. 무결성 (싼 수준)

- **타임스탬프는 클라이언트가 아니라 서버가** 찍는다(`server_timestamp`).
- **append-only 해시체인**: 각 레코드는 직전 레코드의 `record_hash`를 `prev_hash`로
  포함하고, 자신의 정규화 직렬화에 대한 해시를 `record_hash`로 저장한다. 중간 한
  레코드만 고쳐도 이후 체인이 전부 깨지므로 **우발적 단일 수정은 탐지된다.**
- **동시성**: `prev_hash` 읽기 → `record_hash` 쓰기는 단일 트랜잭션 + 직렬화(쓰기 락
  또는 단일 라이터)로 처리해 동시 POST 시 체인 분기·경합을 막는다.
- **보장 한계(명시)**: record_hash는 공개 SHA-256이고 서버 비밀키 서명이 없으므로,
  DB 쓰기 권한을 가진 자가 체인을 `prev_hash`부터 통째로 재계산해 위조하는 **의도적
  전체 재작성은 막지 못한다.** 전체 TSA·WORM·서명은 프로토타입 범위 밖(§10).

## 8. 웹페이지

단일 정적 페이지.

- 파일 업로드 폼 (모드 a 시연).
- 업로드/등록 결과와 `matches` 표 렌더링(파일명·match_type·similarity).
- 등록된 지문·이벤트 목록 표(`GET /api/fingerprints` 결과).

## 9. 검증 시나리오 (이 프로토타입이 통과해야 하는 것)

**테스트 파일 요건**: fuzzy를 검증하는 시나리오(2·4)는 **수 KB 이상의 실제 텍스트
파일**로 수행한다(ssdeep은 블록 크기에 민감해 두 파일 크기가 크게 다르면 유사도가
0으로 나올 수 있으므로, 같은 원본에 소량 편집을 가한 케이스로 본다). 합격 기준은
ssdeep `similarity`(0~100)로 판정한다.

1. **SHA exact**: 같은 파일 두 번 등록 → 두 번째가 첫 번째 baseline과 `exact`,
   similarity 100. (합격: match_type=exact)
2. **fuzzy**: 충분한 크기의 텍스트 파일에서 단어 몇 개 수정 후 업로드 →
   `match_type=fuzzy`이고 **`similarity ≥ 50`**(소량 편집이면 보통 60 이상). (합격:
   fuzzy 매칭 존재 + similarity가 사전 임계치 이상)
3. **이름변경 = 내용 동일(SHA 경로 확인)**: 파일명만 바꾸고 내용 동일 → `exact` 100.
   *(주의: 이건 SHA만 검증한다. fuzzy 단독 동작은 시나리오 2·4가 책임진다.)*
4. **무업로드 트레이스(독립 재계산으로 가설 2 검증)**: (a)로 baseline 등록한 *바로 그
   파일*을, **서버와 독립된 계산**(별도로 호출한 hashlib·ppdeep)으로 다시 계산한
   `sha256`/`fuzzy_hash`를 (b) JSON으로 전송 → baseline과 `exact`(SHA 일치) 그리고
   fuzzy `similarity = 100`(동일 지문 재현). (합격: 본문 없이 보낸 독립 계산 지문이
   서버가 본문으로 계산한 baseline 지문과 재현 일치 → "본문 업로드 없이 트레이스 성립"
   증명. similarity가 100이 아니면 → 이종 계산 간 지문 재현성 문제를 발견한 것이며
   이 또한 가설 2에 대한 유효한 음성 신호다.)
5. **무결성(우발적 단일 수정 탐지)**: 한 레코드의 필드를 임의 수정 후 체인 검증 함수
   실행 → **어느 레코드부터 체인이 깨졌는지**를 반환한다. (합격: 수정된 레코드 지점
   탐지. 이는 우발적 단일 수정 탐지를 시연할 뿐, 의도적 전체 재작성은 범위 밖 §7·§10.)

## 10. 살아남은 우려 (인지하고 진행 — 프로토타입 범위 밖)

- 🔴 신뢰기반: 에이전트는 직원이 끄거나 우회 가능. "억지·감사" 포지셔닝으로 수용,
  타깃 내부자에는 구조적으로 무력.
- 🔴 우회채널: 스크린샷·휴대폰 촬영·클립보드·개인 클라우드 등 파일로 움직이지 않는
  유출에는 장님.
- 🔴 증거능력·합법성(개인정보·근로감시): 실제 배포 시 법무·노사 검토 필요.
- 🟡 파일 이벤트 감지 난이도(move/rename 구분, 임시파일 노이즈, 쓰기 완료 감지):
  클라이언트 구현 단계에서 정면으로 만난다.
- 🟡 정확/바이트-fuzzy 지문의 한계: docx 재저장·포맷 변환·압축·암호화에는 약하다.
  본문 텍스트 추출 기반 문서 지문은 다음 단계.
- 🟡 ssdeep 블록 크기 민감성: 두 파일의 크기가 크게 다르면 내용이 관련 있어도 유사도가
  0으로 나올 수 있다(ssdeep 고유 특성). 같은 원본의 소량 편집 추적엔 충분하나, 큰
  분량 증감엔 약하다. 본문 텍스트 추출 기반 문서 지문(다음 단계)에서 보완.
- 🔴 모드 (b) 지문의 신뢰 한계(R8): 서버는 본문이 없으면 클라가 보낸 지문의 진위를
